# 混合长短请求场景下的大模型推理服务调度优化

## 1. 研究背景与问题定义

本部分工作聚焦大语言模型在线推理服务中的调度优化问题。与离线推理相比，在线服务不仅要求模型能够正确完成 Prefill 与 Decode，还要求在有限资源下兼顾首 Token 时延（TTFT）、整体延迟、吞吐以及显存和 KV Cache 的使用情况。

在混合长短请求同时到达的场景下，长序列请求的 Prefill 更重，更容易占据执行槽位；当服务并发受限时，短序列请求即使本身更轻，也可能被拖慢，最直接的表现就是 short 请求的 TTFT 恶化。该问题属于典型的系统级调度问题，而不是单个算子或单个请求的推理速度问题。

因此，本部分的核心研究问题是：

1. baseline 在线服务在混合长短请求场景下是否存在明显的请求级调度不合理现象；
2. 是否可以通过 request-level 调度优化缓解长序列对短序列的阻塞；
3. 该调度策略能否迁移到更大的 `122B` 模型和 `4xH100` 场景。

从课题要求来看，本部分主要对应两项内容：

- 基于开源推理框架完成在线推理系统搭建与评测；
- 作为开放式研究，定位推理框架在特定 workload 下的性能瓶颈，并给出系统级优化方案。


## 2. baseline 方案与现有机制

本实验基于 `vLLM 0.18.0` 构建在线推理服务。vLLM 已经具备连续批处理、分页式 KV Cache 管理、Chunked Prefill、Decode 优先等机制，这些机制主要作用在执行层和 batch 形成过程，目标是提升通用 serving 效率与资源利用率。

本部分重点观察以下指标：

- `TTFT`
- `平均延迟 / P95 延迟`
- `吞吐`
- `GPU 显存占用`
- `KV Cache 使用率`
- `请求排队与运行状态`

这些指标与课题解读中强调的 `TTFT`、平均延迟、吞吐以及 `KV Cache` 占用比例一一对应。

为了构造混合 workload，实验使用两类提示词数据：

- short 请求：`baseline/prompts.jsonl`
- long 请求：`baseline/test_prompts.jsonl`

并进一步构造 `30% short + 70% long` 的混合集，用于后续对照实验。

### 2.1 vLLM 现有长短相关机制

本文不展开 vLLM 内部实现细节，但需要简要说明其现有长短相关机制。框架侧已有一些与长短序列处理相关的参数与调度手段，例如：

- `enable_chunked_prefill`
- `max_num_batched_tokens`
- `max_num_partial_prefills`
- `max_long_partial_prefills`
- `long_prefill_token_threshold`

这些机制主要发生在 **Prefill / token budget / batch 形成** 这一层，用来优化长请求进入执行之后的推进方式；而本文关注的问题发生在 **请求进入执行之前**，即混合长短请求同时到达时，请求进入执行队列的顺序是否合理。因此，本文的优化重点不在执行层，而在 request-level 排序层。

需要说明的是，本文原本计划进一步测试 `max_num_partial_prefills`、`max_long_partial_prefills` 等“Concurrent Partial Prefill”相关参数，但在本实验实际运行环境中，vLLM 在服务启动阶段直接将该特性判定为 unsupported，并拒绝创建 engine config。因此，本文没有继续沿这组参数展开实验，而改为测试当前环境中可以稳定启用的官方长序列相关机制，例如 `chunked prefill` 与 `max_num_batched_tokens`。这说明：虽然 `vLLM 0.18.0` 已是公开发布版本，但其已发布服务路径中的部分长短相关能力仍未完全开放。

### 2.2 baseline 配置

14B 主实验 baseline 配置如下：

- `MAX_NEW_TOKENS=1024`
- `MAX_NUM_SEQS=64`
- `MAX_NUM_BATCHED_TOKENS=8192`
- `CONCURRENCY=256`
- 混合 workload：`512` 条请求，`154 short + 358 long`

该配置的目的不是压极限吞吐，而是让系统进入“已有明显排队、但尚未完全失控”的区间，从而更容易观察调度策略对 short/long 请求的影响。


## 3. baseline 现象：问题主要出现在排队层

14B baseline 结果表明，系统的主要瓶颈首先体现在**排队**，而不是 KV Cache 先被打满。

### 3.1 14B baseline 关键结果

整体指标：

- 吞吐：`2243.02 tok/s`
- 平均 TTFT：`53899.43 ms`
- P95 TTFT：`80422.76 ms`
- 平均延迟：`77652.62 ms`
- P95 延迟：`106650.16 ms`

资源指标：

- 平均 GPU 显存：`72.04 GB`
- 平均 GPU 显存占用比例：`0.9044`
- 平均 KV Cache 使用率：`0.3137`
- 最大 KV Cache 使用率：`0.4370`
- 平均运行中请求数：`60.46`
- 最大运行中请求数：`64`
- 平均等待请求数：`135.36`
- 最大等待请求数：`192`

可以看到：

- `最大运行中请求数 = 64`，正好打满 `MAX_NUM_SEQS=64`；
- `平均等待请求数 = 135.36`，说明请求竞争主要发生在执行槽位层；
- `平均 KV Cache 使用率 = 0.3137`，远没有达到 KV Cache 饱和区。

因此，在该 workload 下，系统首先进入的是执行槽位竞争和排队放大的区间，而不是 KV Cache 极限区。

### 3.2 short/long 分桶结果

baseline 下，short 与 long 的指标如下：

short：

- 平均 TTFT：`52683.47 ms`
- P95 TTFT：`79488.61 ms`
- 平均延迟：`69545.90 ms`
- P95 延迟：`102541.61 ms`

long：

- 平均 TTFT：`54422.50 ms`
- P95 TTFT：`81266.36 ms`
- 平均延迟：`81139.87 ms`
- P95 延迟：`108200.87 ms`

虽然 short 请求整体上略快于 long，但差异并不显著，说明在 baseline 下，short 请求并没有得到足够强的保护。


## 4. 方法：外部 request-level 粗排

### 4.1 设计思想

本文的核心思路是：**在 vLLM 前增加一个轻量 proxy，在 request-level 做粗粒度排序，再将排好序的流量交给 vLLM 内部 continuous batching 做细粒度调度。**

因此，系统形成了两级调度结构：

1. 外部调度器：决定哪个请求先进入 vLLM；
2. vLLM 内部调度器：决定进入之后如何 batch、如何混合 prefill/decode、如何执行。

这种设计使“请求进入执行队列之前的排序问题”和“请求进入执行之后的 batch 执行问题”得以解耦，从而形成更适合混合 workload 的调度结构。

### 4.2 双队列设计

外部调度器将请求按输入长度划分为两类：

- `short queue`
- `long queue`

划分方式采用 `prompt_chars` 的粗粒度阈值判断，而不是在入口层计算精确 token 长度。其目的不是得到完全精确的作业长度，而是快速识别“明显短请求”和“明显长请求”，从而在 request-level 完成一次低成本粗排。

因此，这里的双队列设计可以理解为：

- **外部粗排**：识别 short/long，请求级别先做排序；
- **内部细排**：vLLM 对已进入请求继续做 batch 级和 token 级调度。

### 4.3 两类关键策略

#### 固定短请求优先调度

该方案采用固定 short 优先策略，验证了“外部双队列 + request-level 粗排”这条路是有效的。

相对 baseline，14B 在该方案下的 short 请求显著改善：

- short 平均 TTFT：`52683.47 -> 12559.43 ms`，`-76.16%`
- short P95 TTFT：`79488.61 -> 27761.16 ms`，`-65.08%`
- short 平均延迟：`69545.90 -> 29467.69 ms`，`-57.63%`
- short P95 延迟：`102541.61 -> 52041.76 ms`，`-49.25%`

但与此同时，long 和 overall tail 明显恶化：

- long 平均延迟：`81139.87 -> 96307.08 ms`，`+18.69%`
- long P95 延迟：`108200.87 -> 124650.20 ms`，`+15.20%`
- overall P95 延迟：`106650.16 -> 123806.55 ms`，`+16.09%`

因此，该方案的意义在于：证明方向成立，但 short-first 过于激进。

#### 比例驱动调度方案

最终形成的 milestone 方案为比例驱动调度。其核心思想是：不再使用绝对等待时间或绝对长队列阈值，而是结合：

- 近期真实到达比例 `arrival ratio`
- 当前队列堆积比例 `queue ratio`

用两者的偏差连续修正 `target_short_share`，从而在保持 short 优势的同时，更自然地回收一部分 long 代价。


## 5. waiting queue 为什么“被挪到外面”

这是本部分必须解释清楚的一点。

### 5.1 baseline 与 proxy 的排队位置不同

baseline 直连链路为：

- `client -> vLLM`

因此，超过后端可承载能力的请求，会直接排在 **vLLM 内部 waiting queue** 中。

而 proxy 版本链路变为：

- `client -> proxy -> vLLM`

此时很多请求会先排在 **proxy 外部队列** 中，只有被 proxy 放行之后才进入 vLLM。

因此：

- baseline 的 waiting 指标主要反映 **vLLM 内部排队**
- proxy 版本的 waiting 指标主要反映 **proxy 外部排队**

### 5.2 证据与对比表

| 指标 | Baseline | FIFO 对照组 | 比例驱动调度 |
|---|---|---|---|
| vLLM 内部平均等待请求数 | 135.36 | 0.1169 | 0.1269 |
| vLLM 内部最大等待请求数 | 192 | 25 | 26 |
| proxy 最大总队列长度 | N/A | 193 | 193 |
| short 平均外部等待时间(ms) | N/A | 52796.39 | 22790.72 |
| long 平均外部等待时间(ms) | N/A | 54512.86 | 64633.23 |
| short 平均外部队列长度 | N/A | N/A | 20.06 |
| long 平均外部队列长度 | N/A | N/A | 110.30 |

这说明：

- 排队并没有消失；
- 排队位置从 vLLM 内部前移到了 proxy 外部；
- 我们真正改变的是排队顺序，而不是消灭排队本身。

进一步看，baseline 的 `avg_num_requests_waiting = 135.36`，而比例驱动调度方案的外部队列平均长度约为 `20.06 + 110.30 = 130.36`，量级非常接近。这说明总排队并没有消失，而是从 vLLM 内部转移到了 proxy 外部；该方案的主要作用是重排外部队列中的 short/long 请求顺序。

### 7.5 小结

同时，FIFO 对照组相对 baseline 基本不变，这说明 proxy 框架本身引入的额外开销较小，比例驱动调度方案的主要变化可以归因于调度策略，而不是 proxy 实现本身。


## 8. 资源、队列与局限

从资源角度看，比例驱动调度方案的收益并不是靠更高显存或更高 KV 占用换出来的。以 122B 为例，相对 baseline：

- 平均 GPU 显存：`74.5671 -> 74.5723 GB`，`+0.01%`
- 平均 GPU 显存占用比例：`0.9362 -> 0.9363`，`+0.01%`
- 平均 KV Cache：`0.3300 -> 0.3304`，`+0.12%`
- 最大 KV Cache：`0.3639 -> 0.3757`，`+3.24%`

变化极小，说明性能变化主要来自调度顺序，而不是资源状态本身变化。

当前方法也有明确局限：

1. long tail 仍然上升，比例驱动调度方案仍然是以 long 代价换 short 收益；
2. waiting 只是前移，不是消失；
3. 最优参数点对 workload 分布敏感。


## 9. 后续方向

当前比例驱动调度方案的主要痛点是：**long tail 仍然明显高于 baseline。**

因此，后续可以在保留当前比例驱动先验控制主循环的前提下，增加慢速后验约束层，例如监控：

- `long_p95_latency`
- `overall_p95_latency`

并先通过冷启动获取当前 workload 下的本地 baseline，再逐步切换到自适应模式，以期在保住 short 收益的同时，将 long tail 控制在更合理的区间内。
