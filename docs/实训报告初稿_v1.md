# 面向混合长短请求场景的大模型推理服务调度优化实训报告（初稿）

## 1. 课题背景与问题定义

本课题聚焦大语言模型推理服务中的系统级优化问题。相比离线推理，在线服务不仅要求模型能够正确完成 Prefill 与 Decode，还要求在有限资源下稳定地兼顾首 Token 时延（TTFT）、整体延迟、吞吐和显存/KV Cache 使用情况。对于边端或资源受限环境，这类问题尤其突出。

本次工作基于 vLLM 0.18.0 构建在线推理服务，先后完成了：

- `Qwen2.5-14B-Instruct` 的在线部署与评测；
- `Qwen3.5-122B` 在 `4x H100` 上的分布式在线部署与评测；
- 面向混合长短请求场景的外部 request-level 调度优化。

本报告重点关注一个具体但很常见的服务场景：**长序列与短序列请求混合到达时，长序列 Prefill 更重、占用执行槽位更久，导致短序列请求被拖慢，用户感知最明显的是 short 请求 TTFT 恶化。**

因此，本工作的核心研究问题不是“如何把单个请求跑得更快”，而是：

1. baseline 在线服务在混合长短请求场景下是否存在明显的请求级调度不合理现象；
2. 是否可以通过 request-level 调度优化缓解长序列对短序列的阻塞；
3. 这种外部调度策略是否能够迁移到更大的 `122B` 模型和 `4x H100` 场景。


## 2. baseline 方案与现有机制

本工作采用 vLLM 0.18.0 作为推理框架。vLLM 本身已经具备连续批处理、分页式 KV Cache 管理、Chunked Prefill、Decode 优先等优化能力，这些优化主要作用于执行层和 batch 形成过程，目标是提升通用 serving 效率与资源利用率。

在本实训中，我们基于其现有服务接口实现完整的在线评测与监控链路，重点观察以下指标：

- `TTFT`
- `平均延迟 / P95 延迟`
- `吞吐`
- `GPU 显存占用`
- `KV Cache 使用率`
- `请求排队与运行状态`

为了构造更接近实际服务的混合 workload，我们使用两类提示词数据：

- short 请求：`baseline/prompts.jsonl`
- long 请求：`baseline/test_prompts.jsonl`

并进一步构造 `30% short + 70% long` 的混合集，用于后续对照实验。

### 2.1 vLLM 0.18.0 现有长短相关机制

虽然本文不做 vLLM 内部实现细节分析，但为了说明本文优化的切入点，仍需简要交代 vLLM 0.18.0 中已经存在的长短相关机制。框架侧已有一些与长短序列处理相关的参数与调度手段，例如：

- `enable_chunked_prefill`
- `max_num_batched_tokens`
- `max_num_partial_prefills`
- `max_long_partial_prefills`
- `long_prefill_token_threshold`

这些机制的共同特点是：它们主要发生在 **Prefill / token budget / batch 形成** 这一层，用来优化长请求进入执行之后的推进方式；而本文关注的问题发生在 **请求进入执行之前**，即混合长短请求同时到达时，请求进入执行队列的顺序是否合理。因此，本文的优化重点不在执行层，而在 request-level 排序层。

需要说明的是，本文原本计划进一步测试 `max_num_partial_prefills`、`max_long_partial_prefills` 等“Concurrent Partial Prefill”相关参数，但在本实验实际运行环境中，vLLM 在服务启动阶段直接将该特性判定为 unsupported，并拒绝创建 engine config。因此，本文没有继续沿这组参数展开实验，而改为测试当前环境中可以稳定启用的官方长序列相关机制，例如 `chunked prefill` 与 `max_num_batched_tokens`。这一点也说明：虽然 `vLLM 0.18.0` 已是公开发布版本，但在其已发布的服务路径中，部分长短相关能力仍未完全开放。

### 2.2 基线配置

14B 主实验使用的 baseline 配置如下：

- `MAX_NEW_TOKENS=1024`
- `MAX_NUM_SEQS=64`
- `MAX_NUM_BATCHED_TOKENS=8192`
- `CONCURRENCY=256`
- 混合 workload：`512` 条请求，`154 short + 358 long`

该配置的目的不是压极限吞吐，而是让系统进入“已有明显排队、但尚未完全失控”的区间，从而更容易观察调度策略对 short/long 请求的影响。


## 3. baseline 现象：问题主要出现在排队层

14B baseline 结果表明，系统的主要瓶颈首先体现在**排队**，而不是 KV Cache 先被打满。

### 3.1 14B baseline 关键结果

整体指标：

- throughput：`2243.02 tok/s`
- avg TTFT：`53899.43 ms`
- p95 TTFT：`80422.76 ms`
- avg latency：`77652.62 ms`
- p95 latency：`106650.16 ms`

资源指标：

- avg GPU mem：`72.04 GB`
- avg GPU mem util：`0.9044`
- avg KV cache usage：`0.3137`
- max KV cache usage：`0.4370`
- avg running requests：`60.46`
- max running requests：`64`
- 平均等待请求数：`135.36`
- 最大等待请求数：`192`

可以看到：

- `最大运行中请求数 = 64`，正好打满 `MAX_NUM_SEQS=64`；
- `avg_num_requests_waiting = 135.36`，说明请求竞争主要发生在执行槽位层；
- `平均 KV Cache 使用率 = 0.3137`，远没有达到 KV Cache 饱和区。

这说明，在该 workload 下，**系统首先进入的是执行槽位竞争和排队放大的区间，而不是 KV Cache 极限区。**

### 3.2 short/long 分桶结果

baseline 下，short 与 long 的指标如下：

short：

- avg TTFT：`52683.47 ms`
- p95 TTFT：`79488.61 ms`
- avg latency：`69545.90 ms`
- p95 latency：`102541.61 ms`

long：

- avg TTFT：`54422.50 ms`
- p95 TTFT：`81266.36 ms`
- avg latency：`81139.87 ms`
- p95 latency：`108200.87 ms`

虽然 short 请求整体上略快于 long，请求级差异并不显著，说明在 baseline 下，**short 请求并没有获得足够强的保护。**

换句话说，vLLM 默认 serving 在该场景下仍然偏向通用执行效率优化，而不是面向混合长短请求场景的 request-level 精细排序。


## 4. 方法：外部 request-level 粗排

### 4.1 设计思想

本工作的核心思路是：**在 vLLM 前增加一个轻量 proxy，在 request-level 做粗粒度排序，再将排好序的流量交给 vLLM 内部 continuous batching 做细粒度调度。**

因此，整个系统形成了两级调度结构：

1. 外部调度器：决定哪个请求先进入 vLLM；
2. vLLM 内部调度器：决定进入之后如何 batch、如何混合 prefill/decode、如何执行。

这种设计的核心价值在于：它把“请求进入执行队列之前的排序问题”和“请求进入执行之后的 batch 执行问题”解耦开来，从而形成两级调度结构。对于本课题而言，这更接近 AI Infra 场景中的调度优化，而不是底层算子优化。

### 4.2 双队列设计

外部调度器将请求按输入长度划分为两类：

- `short queue`
- `long queue`

划分方式采用 `prompt_chars` 的粗粒度阈值判断，而不是在入口层计算精确 token 长度。其目的不是得到完全精确的作业长度，而是快速识别“明显短请求”和“明显长请求”，从而在 request-level 完成一次低成本粗排。

因此，这里的双队列设计可以理解为：

- **外部粗排**：识别 short/long，请求级别先做排序；
- **内部细排**：vLLM 对已进入请求继续做 batch 级和 token 级调度。

也就是说，本文的策略本质上不是替代 vLLM 内部调度，而是在其前面增加一层 request-level admission 与排序逻辑，让后端接收到更有组织的请求流。

### 4.3 从 v1 到 v6 的演进

本项目先后尝试了多个版本。正文只保留对故事最关键的两个阶段：

#### v1：固定 short-first

`v1` 采用固定 short 优先策略，验证了“外部双队列 + request-level 粗排”这条路是有效的。

相对 baseline，14B `v1` 的 short 请求显著改善：

- short avg TTFT：`52683.47 -> 12559.43 ms`，`-76.16%`
- short p95 TTFT：`79488.61 -> 27761.16 ms`，`-65.08%`
- short avg latency：`69545.90 -> 29467.69 ms`，`-57.63%`
- short p95 latency：`102541.61 -> 52041.76 ms`，`-49.25%`

但与此同时，long 和 overall tail 明显恶化：

- long avg latency：`81139.87 -> 96307.08 ms`，`+18.69%`
- long p95 latency：`108200.87 -> 124650.20 ms`，`+15.20%`
- overall p95 latency：`106650.16 -> 123806.55 ms`，`+16.09%`

因此，`v1` 的意义在于：**证明方向成立，但 short-first 过于激进。**

#### v6：比例驱动的折中方案

在经历若干失败迭代后，最终形成 milestone 版本 `v6`。  
`v6` 的核心思想是：不再使用绝对等待时间或绝对长队列阈值，而是结合：

- 近期真实到达比例 `arrival ratio`
- 当前队列堆积比例 `queue ratio`

用两者的偏差连续修正 `target_short_share`，从而在保持 short 优势的同时，更自然地回收一部分 long 代价。

这使得 `v6` 成为目前最好的折中版本。


## 5. waiting queue 为什么“被挪到外面”

这是本实验必须解释清楚的一点。

### 5.1 baseline 与 proxy 的排队位置不同

baseline 直连链路为：

- `client -> vLLM`

因此，超过后端可承载能力的请求，会直接排在 **vLLM 内部 waiting queue** 中。

而 proxy 版本链路变为：

- `client -> proxy -> vLLM`

此时很多请求会先排在 **proxy 外部队列** 中，只有被 proxy 放行之后才进入 vLLM。

因此：

- baseline 的 waiting 指标主要反映 **vLLM 内部排队**
- proxy 版本的 waiting 指标主要反映 **proxy 外部排队**

### 5.2 证据与对比表

| 指标 | Baseline | FIFO Proxy | V6 Proxy |
|---|---|---|---|
| vLLM 内部平均等待请求数 | 135.36 | 0.1169 | 0.1269 |
| vLLM 内部最大等待请求数 | 192 | 25 | 26 |
| proxy 最大总队列长度 | N/A | 193 | 193 |
| short 平均外部等待时间(ms) | N/A | 52796.39 | 22790.72 |
| long 平均外部等待时间(ms) | N/A | 54512.86 | 64633.23 |
| short 平均外部队列长度 | N/A | N/A | 20.06 |
| long 平均外部队列长度 | N/A | N/A | 110.30 |

这说明：

- 排队并没有消失；
- 排队位置从 vLLM 内部前移到了 proxy 外部；
- 我们真正改变的是**排队顺序**，而不是消灭排队本身。

进一步看，baseline 的 `avg_num_requests_waiting = 135.36`，而 `v6` 的外部队列平均长度约为 `20.06 + 110.30 = 130.36`，量级非常接近。这说明总排队并没有消失，而是从 vLLM 内部转移到了 proxy 外部；`v6` 的主要作用是重排外部队列中的 short/long 请求顺序。

以 122B 为例，相对 baseline：

变化极小，说明：

- `v6` 并没有靠更高显存或更高 KV 占用来换性能；
- 性能变化主要来自**调度顺序变化**，而不是资源状态本身变化。


## 9. 本工作的优势与局限

### 9.1 优势

1. **切中真实瓶颈**  
   问题主要发生在请求进入执行队列之前，而本文正是针对该层做优化。

2. **形成两级调度结构**  
   外部负责 request-level 粗排，内部继续负责 batch-level 和 token-level 细排。

3. **能够根据混合 workload 调整排序逻辑**  
   相比默认通用策略，外部调度器更容易显式利用 short/long 比例、队列堆积等信息。

4. **可迁移**  
   `v6` 在 14B 和 122B 上都表现出一致趋势。

### 9.2 局限

1. **long tail 仍然上升**  
   当前 milestone `v6` 仍然是以 long 代价换 short 收益。

2. **waiting 只是前移，不是消失**  
   proxy 版本降低的是 vLLM 内部 waiting，不是端到端等待时间本身。

3. **对 workload 分布敏感**  
   不同 short/long 比例下，最佳参数点会变化。


## 10. 后续方向：v8 设想

当前 `v6` 的主要痛点是：**long tail 仍然明显高于 baseline。**

因此，一个自然的下一步是 `v8`：

- 继续保留 `v6` 的先验控制主循环  
  即使用 `arrival ratio` 与 `queue ratio` 做快速调度决策；

- 在其上增加一个慢速后验约束层  
  例如监控：
  - `long_p95_latency`
  - `overall_p95_latency`

- 先通过冷启动获取当前 workload 下的本地 baseline，再逐步切换到自适应模式。

这样有望在保住短请求时延收益的同时，将 long tail 控制在更合理的区间内。


## 11. 总结

本工作围绕混合长短请求场景，提出并实现了一个基于外部 proxy 的 request-level 调度方案。与默认 baseline 相比：

- `v1` 证明了外部双队列 short-first 路线是有效的；
- `v6` 则进一步在 short 收益与 long 代价之间取得了当前最好的折中；
- 在 `Qwen3.5-122B + 4xH100` 上，`v6` 仍然表现出与 14B 一致的趋势，证明该方法具有较好的可迁移性。

从系统角度看，本文的工作重点不在底层算子，而在**请求进入执行队列之前的调度优化**。通过在入口层增加 request-level 粗排，并与 vLLM 内部 batch-level 细排形成两级调度结构，系统在混合 workload 下获得了更符合场景需求的排队行为。

这说明：**在大模型在线服务中，入口层的请求组织方式本身就是一个值得研究的 AI Infra 优化点。**
