# 面向大语言模型高效推理框架的系统级优化与实现：完成规划

## 1. 课题本质

这不是一个“把模型跑起来”的题目，而是一个“围绕 LLM 推理系统做工程化优化并给出实验论证”的题目。  
你们最终要交付的，应该是一个可运行、可测量、可对比、可讲清楚原理的大模型推理系统原型。

结合 `实训赛题.md`、`课题解读.pptx` 与 `baseline/README.md`，该课题的核心要求可以归纳为：

- 跑通完整推理链路：模型加载、Tokenizer、Prefill、Decode、生成输出。
- 基础功能最好覆盖量化、流式输出；PPT 中还提到多模型格式支持，至少需要在报告里说明实际支持了哪些格式、哪些没有做。
- 在 baseline 基础上提升效率：重点看 TTFT、平均/分位延迟、吞吐率、显存占用。
- 保持精度：优化后精度损失不宜超过 5%。
- 形成完整交付：代码、运行说明、实验结果、实验报告、展示材料。

## 2. 推荐技术路线

### 主路线

推荐采用：

- 模型：`Qwen2.5-14B-Instruct` 及其量化版本
- 推理框架：`vLLM`
- 对比基线：当前 `baseline/` 中基于 Hugging Face `generate()` 的朴素实现

### 为什么推荐 `vLLM`

- 题目环境是单卡 H100，更适合走 GPU 推理服务优化路线。
- 题目强调动态 batch、KV Cache 管理、PagedAttention 或等效机制，这些正是 `vLLM` 的强项。
- 时间有限，从零手写推理框架风险很高；基于成熟框架做系统封装、评测与优化，更容易完成高质量交付。

### 备选路线

`llama.cpp` 可作为补充路线，用于：

- 做低资源部署对比
- 展示量化和 CPU/边端部署思路
- 在报告中作为“不同技术路线的比较”

但如果目标是稳定完成课题并争取主要指标，主线仍建议放在 `vLLM`。

## 4. 建议的完成阶段

### 第一阶段：跑通 baseline

目标：先拿到一组可信的 baseline 数据。

要做的事：

- 安装依赖，准备模型权重。
- 确认实验环境按 `单卡 cuda:0` 统一，避免后面对比口径不一致。
- 运行 `baseline_inference.py`，确认模型能正常加载和生成。
- 运行 `benchmark.py`，得到 `results_baseline.json`。
- 运行 `evaluate_accuracy.py`，得到 `accuracy_baseline.json`。
- 如果精度评测数据量过大，可固定抽取同一批 200 到 500 条样本，保证优化前后使用完全一致的评测集。

产出：

- baseline 运行截图或日志
- `results_baseline.json`
- `accuracy_baseline.json`

### 第二阶段：搭建优化版推理路径

目标：从“单条朴素推理”升级到“具备系统优化能力的推理服务”。

建议优先实现这些能力：

1. 使用 `vLLM` 跑通同一模型
2. 支持流式输出
3. 支持批量请求或连续 batching
4. 明确支持的模型格式范围，并在 README/报告中写清楚
5. 对接统一 benchmark 接口
6. 对接统一 accuracy 接口

建议的代码组织：

- `vllm_base/engine.py`：封装 vLLM 基础版推理引擎
- `vllm_base/inference.py`：单次推理与流式演示
- `vllm_base/benchmark.py`：vLLM 基础版性能评测
- `vllm_base/evaluate_accuracy.py`：vLLM 基础版精度评测

### 第三阶段：做“有证据”的优化

不要一次上太多改动。推荐按可量化收益逐项推进：

1. `vLLM` 替代 HF baseline
2. 启用/验证 Continuous Batching
3. 使用量化模型版本，比较显存与吞吐
4. 改进 TTFT 统计方式，真正记录首 token 时间
5. 增加 KV Cache / Prompt Cache 复用实验

每做完一项都保存结果，形成对比表。

### 第四阶段：实验对比与消融

报告里必须让老师看到“优化前后差异”，所以至少做三类实验：

1. baseline vs optimized
2. full precision vs quantized
3. 是否开启某项优化（如 batch/cache）的对比

建议记录的表格字段：

- `overall_throughput_tps`
- `avg_latency_ms`
- `p95_latency_ms`
- `average_ttft_ms` 或等价字段
- `peak_gpu_mem_gb`
- `accuracy`
- 如能拿到，更建议补充 `KV Cache` 占用比例或缓存命中情况；如果拿不到，也应在报告中说明原因。

### 第五阶段：报告与答辩材料

实验报告建议按这个结构写：

1. 课题背景与目标
2. baseline 分析
3. 系统架构设计
4. 关键优化点
5. 实验设置
6. 结果对比
7. 性能瓶颈分析
8. 不足与未来工作

另外建议单独加一小节：

9. AI 工具辅助说明

这里写清楚是否使用了 `Claude Code`、`Cursor` 等工具，以及它们分别帮助了哪些环节。PPT 里明确提到这点，最好不要漏。

答辩时要能讲清楚这些概念：

- Prefill 和 Decode 的区别
- KV Cache 的作用
- PagedAttention/分页管理为什么省显存
- 动态 batch 为什么能提吞吐
- 量化为什么能降显存、有什么精度代价
- 你们的系统瓶颈在哪里，怎么定位出来的

## 5. 建议的任务分工

这是三人课题，建议按“系统实现 / 评测实验 / 文档答辩”拆分，但不要完全割裂。

### 成员 A：推理引擎与服务封装

- 跑通 `vLLM` 或其他主路线
- 封装 optimized inference
- 做流式输出与接口统一

### 成员 B：评测与优化实验

- 维护 benchmark / accuracy 脚本
- 跑各组实验
- 汇总性能、显存、精度结果

### 成员 C：报告、图表与技术分析

- 负责 baseline 分析和优化说明
- 维护实验表格与图
- 整理 README、实验报告、答辩 PPT

同时三人都要能讲清楚完整流程，避免答辩时只有一个人懂系统。

## 6. 低风险必做项与加分项

### 低风险必做项

- 跑通 baseline
- 跑通 optimized 路线
- 生成 baseline/optimized 的性能与精度 JSON
- 支持流式输出演示
- 写出系统架构设计说明
- 写出结构清晰的 README 与实验报告

### 建议优先争取的加分项

- 量化实验
- 动态 batching 对比
- KV Cache / Prompt Cache 复用实验
- 明确的瓶颈分析过程

### 风险较高、放在最后

- 自研 PagedAttention 底层实现
- 超大模型分布式部署
- 复杂 CUDA 级算子优化

## 7. 当前最合理的下一步

如果你们现在马上开始做，建议按下面顺序执行：

1. 先完整复现 baseline，拿到两份 baseline JSON。
2. 确认模型权重路径、CUDA 环境、显存占用是否正常。
3. 选定主路线为 `vLLM`，先实现一个最小 optimized 推理脚本，并明确支持的模型格式范围。
4. 让 optimized 脚本也能产出与 baseline 一致格式的结果文件。
5. 在此基础上逐项加入 batching、流式输出、量化和 cache 复用。
6. 从一开始就同步整理架构图、实验记录和 AI 工具使用说明，避免最后补文档。

## 8. 一句话结论

最稳的完成方式不是“自己写一个全新推理框架”，而是“基于 baseline 建立统一评测口径，再用 `vLLM` 做系统级优化与实验对比”，把代码、指标、报告和答辩材料一起做完整。
